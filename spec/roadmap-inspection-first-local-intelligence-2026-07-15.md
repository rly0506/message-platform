# RM-065 候选路线图：检视阅读优先的本地智能工作台

> 路线图编号：`RM-065`。状态：`CANDIDATE`。本文件记录人类已确认的方向与实施闸门，**不取代**当前唯一主线 RM-055，也不自动授权实现、真实库运行、扩源、合并或发布。

## 一句话方向

先把历史遗留和正确性边界说清，再用“检视阅读”帮助用户快速筛选和粗略了解材料；随后用实测性能、本地聚类质量和 Coverage 缺口逐级决定优化与扩源，不把“更先进算法”或“更多来源”本身当作成果。

## 人类已锁定的选择

- 推进结构：**证据阶梯**。
- 候选优先级：**检视阅读 → 性能优化 → 本地分析加强 → 信息源拓展**。
- RM-065 先保持 `CANDIDATE`，RM-055 继续作为唯一 `CURRENT`。
- 探针短段仅在本地临时保存，Grok 审计后删除；长期证据只保留指标、哈希、URL 和失败分类。
- 产品化后使用独立本地检视材料库，默认保留 90 天，可刷新、可删除。
- 每次成功加载话题后最多预取 3 篇最新相关报道，其余文章由用户点击后抓取。
- 本地分析首个目标是**事件聚类与转载去重**。
- 商业模式画布、前景理论只是可替换分析框架的例子，不是既定首批镜头。

## 证据阶梯

### H0 · 历史真相与文档收口

**GPT-5.6**：形成未完成事项处置表；修正 `spec/roadmap.md` 对观察管线“待实施”的过期描述；完成 E2/E3/E4/C 文档卫生。

**Grok**：对照 ledger、current-state、提交和旧路径，确认没有把完成项重开。

**出口**：CURRENT、ACTIVE-GATE、CANDIDATE、COMPLETED/SUPERSEDED 四类无冲突；Markdown 链接、UTF-8 和 `git diff --check` 通过。

### G0 · RM-055 观察闸持续运行

保持现有 HTTP-first、fail-closed 和 `fulltext unknown`。只有在人类另行授权真实库运行且后端可达时，才执行标准 `refresh-once` 并记录成功或失败。检视、性能和本地分析都不得制造观察成功日或扩源理由。

### P0 · 话题加载正确性

**Opus 4.8**：为全部话题级异步加载增加单调 revision/取消保护；确定一个“选题后加载”的唯一所有者；搜索/深度任务完成后不得重复请求已经由该所有者加载的资源。

**公共接口**：不改后端 API/DTO。

**出口**：在 A 请求故意延迟、用户快速切到 B 的测试中，topic、articles、local events、event graph、academic、sentiment、cross-synthesis 最终全部属于 B；每种资源最多一次有效加载。

### R1 · 检视阅读证据探针

**GPT-5.6**：从文章数最大、中位、最小非空的三个活跃话题中，各选最新 10 个已入库公开 URL。只做瞬时抓取，提取原链、标题、发布方摘要、首段、尾段、状态、耗时及逐字段 provenance；无 LLM、不写 `dossier.db`、不接 UI、不启用新源。

探针短段写入 gitignored 临时运行目录，供 Grok 抽检；审计结束后删除。长期报告不含段落正文，只保留每 URL 状态、哈希、指标、失败分类和原链接。

**出口**：30/30 URL 有明确终态；合格公开页面可用率至少 70%；Grok 抽检至少 10 条且至少 9 条无导航、评论或页脚污染；清理后没有段落文本残留。未达门槛则 `DEFER`，不引入重爬虫补救。

### P1 · 测量驱动的性能优化

**GPT-5.6**：使用临时 SQLite 的固定工作负载（6,702 Article、875 个话题关联、128/465 incoming）逐项评估：

1. `collect_topic` 只投影 URL、title 和必要关联，不物化无关 ORM 列；
2. 预计算标题 key 与查询词，避免重复规范化和列表拼接；
3. 验证 `flush()` 回填 ID 后移除冗余 `refresh()`/逐项关联查询；
4. 将自动刷新最新发布日期合为一次 SQL 聚合，但保持空话题、排序和 cycle limit 语义。

**公共接口**：无 API、DTO 或 schema 变化。

**出口**：kept URL、relevance、Article/TopicArticle 数据逐项等价；p95 至少改善 20%；SELECT 数至少下降 30%；峰值内存不得恶化超过 10%。任一候选不达门槛就不合入。

### P2 · 检视阅读产品化

#### 后端契约

新增独立 `ArticleInspection` 本地表，不给 `Article` 增加正文列：

- `article_id`、`status`、`source_url`；
- `lead`、`tail`（各最多 600 字符）；
- 每字段 provenance、`fetched_at`、`expires_at`；
- `content_sha256`、`error_code`。

发布方摘要继续来自 Article 现有 snippet，API 最多返回 600 字符，不重复保存。记录默认 90 天过期；读取、刷新和既有每日任务顺带清理；文章删除时同步删除。用户可以立即刷新或删除。

新增 additive 接口：

- `GET /api/topics/{topic_id}/articles/{article_id}/inspection`：只读有效缓存；缺失返回 `missing`，不发网络；
- `POST .../inspection`：按需抓取或主动刷新，返回检视材料；
- `DELETE .../inspection`：删除本地材料。

抓取失败返回可展示的 `unavailable/blocked/partial`，不让核心阅读失败。只访问普通公开 HTTP(S) 页面；明确 robots 禁止、登录、付费墙、验证码、401/403/429 均停止，不使用指纹伪装或对抗式抓取。

#### 前端体验

**Opus 4.8**：文章列表仍先正常显示。每次当前话题文章成功加载后，对排序靠前且 `relevant` 的最多 3 篇发起非阻塞、并发受限的检视请求；其余文章在用户点击“检视阅读”时抓取。

检视卡明确展示：发布方摘要、正文首段、正文尾段、原文链接、抓取时间、字段来源和“检视阅读用于筛选，不代表已读全文”。现有需要 LLM 的“透视”功能保持独立，不复用或混名。

**出口**：预取永不超过 3 篇；第四篇只在点击后请求；切换话题时迟到结果被丢弃；网络失败不阻塞列表；桌面和移动端均可访问。

### P3 · 本地事件聚类与转载去重

**GPT-5.6** 先建立冻结的同事件/不同事件标注集，覆盖中英文、同源转载、相关但不同事件和时间跨度。Grok 复核标签后记录当前 RapidFuzz 基线。

只有基线出现具名质量或比较数问题时，才实现“45 日时间窗口 + token 倒排候选 → 现有 RapidFuzz 精排”。MinHash/LSH 只有在该方案仍不达标时进入对照；ML 只有在固定训练/验证/时间外测试集上显著超过规则基线时才可进入产品。

**出口**：正例候选召回至少 99%；错误合并不高于基线；候选比较数下降至少 70%；p95 至少下降 40%；任一语言切片质量下降不得超过 2 个百分点。未达门槛则保留现算法并记录 `NO-GO`。

检视材料暂不直接改变 Event/EventRelation。事件图消费检视证据前，必须另行解决本地抽取与 LLM 深度分析的 source-of-truth。

### P4 · 证据驱动的信息源拓展

**GPT-5.6** 先补齐 15 个已启用源的 coverage/access metadata debt，不把 metadata unknown 当成来源缺口。

RM-055 闸门通过后，只针对一个具名、重复出现的语言/国家/角色/来源层级缺口，按 `RSS/Atom → Sitemap → 有界公开页` 选择首批最多 3 个来源。每个来源必须跨三日成功、URL 稳定、无重复洪水并改善原缺口。

新来源类型优先公开个人网站、博客和 Newsletter；其次是公开 transcript。LinkedIn 只作人工确认的公开线索，不进入自动登录抓取；私域资料必须另立授权导入路线。

## 统一验证与审计

每一阶段都执行：父代理先列清单 → 派独立子代理 → 父代理亲自复核 → TDD/最小实现 → Grok 独立审计 → 目标门禁 → GitNexus `detect-changes`。

- 后端：临时数据库专项测试后跑全量 pytest；不得写真实 `backend/dossier.db`。
- 前端：focused Playwright、构建、桌面/移动端完整 E2E。
- 性能：记录 p50/p95、SQL 数、比较数和峰值内存，不只报告理论复杂度。
- 数据边界：`ArticleInspection` 永不存完整正文；Coverage `fulltext.status` 继续为 `unknown`。
- Git：每阶段独立提交；未经人类明确授权不合 master、不发布。
