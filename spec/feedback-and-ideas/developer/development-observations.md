# 开发者观察

Status: `DEVELOPER_NOTE`

本文件保存开发者对用户意见的理解、技术判断和分歧。它不能覆盖用户原始意见，也不能单独授权开发。

## 当前共识

### 意见资料库的边界

- 用户意见和感悟应被长期保存，即使暂时无法实现。
- 开发者判断必须注明作者、日期和依据。
- 参考项目应记录"借鉴、暂缓、拒绝"三个层次，避免整套照搬。
- 路线图只接收经过人类明确批准的任务。

### 知识图谱方向

- 项目真正需要的不是视觉上复杂的大图，而是事件、实体、关系和来源之间可验证的联系。
- 应先用现有 SQLite 结构验证一个无 LLM 的多跳证据问题。
- 关系至少区分：原文直接观察、规则推导、LLM 或分析者假设。
- 每个多跳结论必须能展示路径和来源；不能把不同性质的关系压成单一可信度。
- 只有现有关系模型出现可测量的复杂度或性能瓶颈后，才重新评估图数据库、OpenSPG 或 KAG。

### 静态知识发布方向

- 私有工作台与公开展示层应分离。
- 对外发布需要白名单、构建期清洗和明确的隐私边界。
- 静态搜索适合导出的只读资料，不适合实时任务、私有数据和可变业务状态。
- 自动生成内容必须幂等、可追溯并允许在没有 LLM 时跳过。

## 信息供应链诊断（Claude，2026-07-12；不代表用户原话）

### 起因

用户指出：经历几轮"扩源"迭代后，产品实际打开看到的仍是原来的 RSS、Google News 摘要和少量社区内容，没有摆脱困境。GPT 的只读审计佐证了这一点。

### 核心判断

- 症结不是"源不够多"，而是**增强能力的代码和开关已经做好，但默认全关、没有真实数据验收、界面也不显示本轮用了哪些能力**。SearXNG / gnews 解码 / Scrapling 三条路径默认 `False`，产品路径吃到的仍是 RSS + 摘要。
- 因此路线图看起来"完成"，与实际体验脱节——"代码存在"被当成了"产品拥有能力"。
- 更深一层是用户的汉弗莱担忧：用户怕的不是 AI 读得少，而是 AI 读了什么、漏了什么、有没有反证，用户看不见，只能说 yes。

### 建议顺序（A/B/C）

- **C 先行**（让每个 AI 结论都能看到证据与反证 + 覆盖率仪表），因为 A（扩源）和 B（摘要→正文）若先做，只会把更多材料塞进一篇更流畅的总结里，反而喂大汉弗莱。
- C 的真身不是一个 UI 功能，而是**我们一直缺的验收仪表**：真实全文成功率、独立原始源比例、多语言覆盖、一手资料比例、同源转载去重率、关键角色覆盖率。
- 一旦 C 建好，它同时是：给用户的诚实覆盖标签、给 A/B 的验收标准、逼我们打开现成开关的动力。
- C 的第一个动作就顺手把 `GNEWS_DECODE_URLS`、Scrapling 等现成开关打开，用最便宜的 B 给 C 喂第一口真实数据。

### 与门禁的关系

- 过去的验收偏工程（接口没崩、开关有效、mock 能解析、前端能显示），从未验证信息质量（真实成功率、独立源比例、多语言覆盖、一手资料比例、去重率、角色覆盖率）。
- 这是"几轮迭代没突破"的流程性原因，也是 reviewer 复验承重改动时要加的一问：不只问"测试绿吗"，还要问"这条能力默认开着吗、真实数据上量到了吗"。
