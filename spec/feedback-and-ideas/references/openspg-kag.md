# OpenSPG/KAG 参考记录

Checked: 2026-07-12

Status: `REFERENCE`

## 项目

- 代码：<https://github.com/OpenSPG/KAG>
- 论文：<https://arxiv.org/abs/2409.13731>

## 核心价值

KAG 基于 OpenSPG 和大语言模型，面向垂直领域知识库的检索、逻辑推理和事实问答。它的重要启发不是“必须换成图数据库”，而是让知识结构、原始文本和推理步骤保持可互相追溯。

## 值得借鉴

- 在结构化事件、实体、关系与原始文章/证据片段之间建立双向索引。
- 使用显式关系模式，避免任意、无来源的图边。
- 把精确查找、文本检索、图遍历、计算和可选 LLM 推理标成不同步骤。
- 多跳回答展示完整证据路径，而不是只输出综合结论。
- 区分原文观察、规则推导和模型假设。

## 当前不直接集成的原因

- OpenSPG/KAG 的平台、容器和模型依赖明显重于当前 SQLite 架构。
- 将其放入核心路径会增加部署与运维负担，并可能破坏无 LLM 可运行原则。
- 论文和项目演示中的效果不能替代本项目数据上的验证。
- 当前更需要确认用户是否真正使用事件关系和多跳证据能力，而不是先搭建大型图平台。

## 推荐验证顺序

1. 先用现有 SQLite 事件、实体、关系和文章 ID 回答一个有限的多跳问题。
2. 展示每一跳的关系类型、来源和生成方式。
3. 记录查询复杂度、正确率、人工理解成本和性能。
4. 只有出现可测量瓶颈，才评估图数据库或 KAG/OpenSPG。

## 关系类型底线

| 类型 | 含义 |
| --- | --- |
| `OBSERVED` | 原始来源直接陈述或结构化数据明确证明。 |
| `DERIVED` | 通过透明规则、时间或实体匹配计算得到。 |
| `HYPOTHESIS` | LLM 或分析者提出、仍需验证的假设。 |

三类关系不能混成一个统一可信度，也不能让 `HYPOTHESIS` 在界面上伪装为事实。
